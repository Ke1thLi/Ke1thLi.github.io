前言

作为一个智能科学与技术专业大一的学生，同时也在学校网安部参与技术学习，我一直想搭建一个属于自己的技术博客。经过各种权衡，我最终确定了 Hugo + GitHub Pages + Obsidian 的技术栈。

在这个过程中，我不仅对比了各种工具的优缺点，还结合了自己的专业背景做了很多考量。作为一个网络安全方向的学生，我在技术选型时会格外关注数据安全和可控性——这也是这篇文章和其他"博客搭建教程"不太一样的地方。

这篇文章记录了我整个技术选型的过程，以及为什么放弃了其他热门方案（如 Hexo、Quartz、单仓库部署、图床等）。

这篇文章不会教你“如何一步步搭建”，而是分享我 为什么这样选。如果你也看重 安全、长久可维护、数据自主，低心智负担或许能给你一些参考。

一、为什么是 Hugo？—— 告别 Node.js 依赖焦虑

静态站点生成器有很多，我重点对比了 Hugo、Hexo 和 Quartz。

维度Hugo (Go)Hexo (Node.js)Quartz
依赖单个二进制文件，无运行时依赖npm 生态，Node 版本容易冲突早期版基于 Hugo，后来部分分支自研，依赖复杂
构建速度极快（毫秒级），文章数量几乎不影响文章多了明显变慢取决于底层实现，一般中等
10 年后还能跑吗？✅ 下载对应系统的二进制，即下即用⚠️ 需要搞定 Node 版本和 node_modules，容易出问题❓ 未知，生态较小
主题生态丰富，很多高质量开源主题也非常丰富较少，主要偏向“数字花园”风格
学习曲线较低，模板语法简单中等中等

我最在意的是 长期可构建性。在网络安全领域，我们常说“没有绝对的安全，只有持续的风险管理”。对我来说，Hexo 的 Node.js 依赖链就是一种“长期技术债”——你永远不知道几年后 npm install 会报什么错。而 Hugo 只是一个可执行文件，下载、解压、放到 系统$PATH环境变量。

我也看了 Quartz，它主打“Obsidian 数字花园开箱即用”。但它的定制灵活性不如原生 Hugo，且部分功能依赖额外的 Node 工具链。我更希望博客的外观和扩展能力完全由自己掌控，而不是被一个特定框架限制。

对于我而言，Hugo可能时是目前最好的选择。

二、为什么选择 Obsidian 作为写作工具？

我身为一名学生，在自己本地笔记仓库包含学习的笔记，ctf，oi，靶场渗透的记录，甚至还有一些读书笔记。

知识点之间并不是孤立的，而是相互关联的。Obsidian 的 双向链接（[[ ]]）是吸引我的主要原因，我可以把这些知识点串联起来，但 Obsidian 的一些“增强语法”与 Hugo 不兼容。

遇到的坑与解法：

  1. Wiki 链接（[[另一篇笔记]]）
    → 关闭设置中的“使用 WikiLink”，改用标准 Markdown 链接。

  2. 图片语法（![[图片.png]]）
    → 改用 ![描述](./图片.png)（相对路径）。配合插件 Paste Image Rename 可以自动转换。

  3. Dataview 查询
    → 个人笔记库随便用，但发布到博客前，我把 Dataview 生成的表格/列表手动复制为静态文本。博客不需要动态查询，静态内容更安全。

  4. Frontmatter 格式
    → 用 Templater 插件统一生成 Hugo 要求的 YAML 格式：

6

    ---
title: 文章标题
date: 2026-04-27
tags: ["CTF", "Web安全"]
draft: false
---

为什么不单独用 Typora？
Typora 的沉浸式写作体验很好，但我更希望保持单一的笔记系统。Obsidian 虽然编辑体验稍弱，但它的知识管理能力是 Typora 不具备的。我在 Obsidian 里写好草稿，最后用 Typora 做一次预览润色，两者互补。

三、为什么选择双仓库方案？

在部署方案上，我选择了双仓库架构：一个私有仓库存源码，一个公开仓库放生成的静态文件。虽然单仓库方案更简单，但双仓库对我来说有几个不可替代的优势。

2. 双仓库的优势

源码隐私保护

作为学生，我有些笔记可能包含：

  • 未完成的草稿和实验记录
  • 某些不想公开的配置文件

私有仓库存源码让我可以放心写草稿，不用担心未完成的内容被公开。只有构建后的静态文件（即我确认发布的文章）才会出现在公开仓库里。

部署仓库干净

公开仓库只包含生成的 HTML、CSS、JS 文件：

  • 没有 Markdown 源码
  • 没有配置文件
  • 没有主题代码
  • 体积小，结构清晰

如果有人想参考我的博客部署方式，看到的只是干净的静态文件，不会被复杂的配置干扰。

灵活的多平台部署

双仓库架构让我以后可以轻松扩展到多个部署平台：

平台用途
GitHub Pages主要部署，国内可访问
Netlify备用部署，全球 CDN 加速
Cloudflare Pages国内优化访问

只需要在 blog-source 的 GitHub Actions 里添加部署步骤，就可以同时推送到多个平台。单仓库方案要做多平台部署会复杂得多。

3. 单仓库适合谁？

场景说明
纯公开博客所有内容都想开源
简单个人站不需要隐藏任何配置
新手入门想最快跑起来，不想折腾

4. 我的考量

选择双仓库确实增加了配置复杂度，需要理解：

  • GitHub Actions 的跨仓库推送
  • Deploy Key 的配置
  • CI/CD 流程的设计(部署这个博客时，拿ai完成的这个)

理解这些部署流程本身就是一种学习。而且一次配置好后，日常写博客的体验和单仓库没有任何区别——都是 git push 就完事了。

双仓库给了我隐私保护和部署灵活性，让我可以专注于写内容，不用担心源码公开的问题。

四、为什么不设置图床？

这是我纠结最久的一个决定。网上很多教程都推荐配置 PicGo + 图床，但我最终选择了本地存储图片。原因如下：

1. 图床方案对比

方案优点缺点
PicGo + GitHub免费、国内可访问GitHub ToS 灰色地带、图片公开
PicGo + Cloudflare R2流量免费、速度快配置复杂、需要信用卡
本地存储安全合规、零配置仓库体积会增长、图片不公开

2. 网安笔记的敏感性

作为网安社团成员，我经常需要写 CTF 靶场复现和漏洞分析。这些文章里不可避免地会包含：

  • 敏感接口响应数据
  • Cookie / Token 信息

如果使用图床，这些图片会被上传到第三方服务器。即使我手动打码，也存在漏打的风险。而本地存储可以确保敏感图片只存在于我的电脑中。

3. GitHub ToS 的潜在风险

GitHub 的服务条款虽然没有明确禁止将仓库用于图片存储，但也指出：

Repositories may not be used for unauthorized file hosting or CDN purposes.

这意味着如果你的仓库主要是大量图片而非代码，可能会被认定为滥用。虽然个人博客的少量图片通常不会触发限制，我倾向于选择完全合规的方案，避免任何潜在风险。

补充：如果你决定使用 GitHub 做图床，建议控制图片数量（单张图片 < 1MB）和仓库总体积（建议 < 1GB），避免被限制。

4. 长期维护的可靠性

图床服务可能会：

  • 停止运营：很多免费图床说关就关，之前的图片全部失效
  • 改变政策：免费额度缩减、开始收费
  • 访问受限：部分国外图床在国内访问不稳定

唯一代价： 仓库体积会稍大。但我估算过，就算写 5 年、300 篇文章、每篇配 5 张 300KB 的截图，总大小也不到 500MB。GitHub 仓库建议 1GB 以内，完全够用。

五、总结

技术选型选择理由
静态生成器Hugo构建快、单文件、生态成熟
部署平台GitHub Pages免费、自带 HTTPS、Actions 自动化
写作工具Obsidian双向链接、本地优先、插件丰富
图片存储本地跟随笔记安全合规、工作流简单、够用
仓库架构双仓库源码隐私、部署干净、多平台扩展
自动化部署GitHub Actionsgit push 即可更新、云端构建、无需本地环境
我的诉求方案如何满足
安全可控无图床 + 本地脱敏 + 私有源码仓库（发布前可审核）
长久可维护Hugo 二进制 + 纯 Markdown + Git 版本控制，10 年后依然可构建
低心智负担没有 API Key、没有防盗链配置、没有账单管理
写作流畅Obsidian 写作 + Typora 预览 + 少量兼容性调整，图片自动保存到文章同级目录

最终，我的博客工作流非常朴素，也是我长期主义的体现：

  1. 在 Obsidian 的知识库里写草稿，尽情用 Dataview、双向链接。
  2. 把要发布的文章复制到 Hugo 项目的 content/posts/ 下，调整 frontmatter，图片用本地相对路径。
  3. 用 Typora 打开预览，检查格式和图片。
  4. 运行 git add . && git commit -m "new post" && git push origin main。
  5. GitHub Actions 自动构建并部署到 ke1thli.github.io。

没有复杂的图床上传，没有担心流量账单，没有“图片裂开”的焦虑。

这套技术栈可能不是最"酷"的，但对我来说是最稳、最可控的选择。
